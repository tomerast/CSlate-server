## 1. @cslate/shared Package — Who Owns It?

**Agree: Own repo (`CSlate-shared`).** Neither client nor server should own the shared types — changes affect both equally.

**Structure:**
```
@cslate/shared
├── src/
│   ├── schemas/
│   │   ├── manifest.ts           # ComponentManifest Zod schema (source of truth)
│   │   ├── search.ts             # SearchRequest, SearchResponse schemas
│   │   ├── upload.ts             # UploadRequest, UploadResponse schemas
│   │   ├── checkpoint.ts         # Checkpoint schemas
│   │   ├── errors.ts             # ApiError schema + error codes enum
│   │   └── auth.ts               # Auth request/response schemas
│   ├── types/
│   │   └── index.ts              # Inferred types from Zod schemas (z.infer<>)
│   └── index.ts                  # Barrel export
├── package.json
└── tsconfig.json
```

**Workflow:**
- Any manifest or API contract change starts as a PR to `CSlate-shared`
- Both repos pin to a specific version (not `latest`)
- Breaking changes require coordinated releases (shared bumps major → both repos update)
- The Zod schemas are the **single source of truth** — runtime validation + TypeScript types + (via `@hono/zod-openapi`) OpenAPI docs, all from one definition

**Publishing:** npm (scoped `@cslate/shared`) or git submodule. npm is cleaner for versioning. For early development, a git submodule or workspace symlink works too.

**Server action:** We'll create the initial `CSlate-shared` repo with the reconciled manifest schema once the contract is finalized.

## 2. context/decisions.md — What the Server Does With It

**Agreed on all three points. Here's the full policy:**

### What the server DOES:
- **Preserves it exactly as uploaded.** Zero modification. It's the user's conversation history.
- **Indexes key terms for search relevance.** During the embedding stage, the server extracts salient terms from `decisions.md` and includes them in the composite embedding. This means if a user built a "todo list with drag-and-drop reordering because their team uses Kanban boards," future searches for "kanban" or "drag reorder" will surface this component even if those words aren't in the manifest description.
- **Surfaces it in search results.** The `SearchResponse` will include a `contextSummary` field — a 1-2 sentence AI-generated summary of decisions.md that explains "why this was built." This helps the client AI agent decide if a blueprint's intent matches the user's need, not just its surface-level description.

### What the server does NOT do:
- **Does not reject based on context.md content.** If the user's conversation context is messy, profane, or contradictory, that's fine — it's not code.
- **Does not modify it.** Not even to clean up formatting.
- **Does not use it to second-guess the code.** Stage 4 (Context Verification) checks that the *code* matches the *stated requirements* — it doesn't reject because the conversation was indecisive. It flags discrepancies only when the code clearly contradicts explicit requirements stated in decisions.md (e.g., decisions.md says "must support dark mode" but the code has no dark mode handling).

### Addition to SearchResponse:
```typescript
interface SearchResult {
  // ... existing fields ...
  contextSummary?: string;   // AI-generated 1-2 sentence summary of WHY this was built
}
```

## 3. ai.modificationHints and ai.extensionPoints — Generation Timing

**They are available immediately when the component is approved.** Not async after approval.

Here's why: The review pipeline is already async (1-3 minutes). Stage 5 (Manifest Enrichment) generates the `ai` fields BEFORE Stage 7 (Cataloging). By the time a component status changes to `approved`, the manifest already contains the enriched `ai` field.

**Timeline within the pipeline:**
```
Stage 1-4: Validation + Review (~30-90s)
Stage 5:   Manifest Enrichment — ai.modificationHints, ai.extensionPoints generated HERE
Stage 6:   Embedding generation
Stage 7:   Cataloging → status = "approved" → component goes live with enriched manifest
```

**What the client can rely on:**
- Any component returned by `GET /api/components/search` or `GET /api/components/:id` will ALWAYS have the `ai` field populated
- The `ai` field is part of the manifest returned in search results, so the client AI agent can evaluate modification guidance BEFORE fetching the full source
- If the original uploader's client polls for status after approval, the returned `manifest` will include the enriched `ai` field

**One edge case:** The `ai.modificationHints` are generated by the server's LLM based on code analysis. They may occasionally be imperfect. The client AI agent should treat them as helpful guidance, not absolute truth. If a hint says "modify file X" but file X doesn't exist in the package, the client AI should ignore that hint.

## 4. Component Versioning — Update Notifications

**Yes, we should notify users of available updates.** Here's the design:

### New endpoint:
```
POST /api/components/check-updates
```

**Request:**
```typescript
{
  componentIds: string[];    // Server IDs of components the user has on their Slate
}
```

**Response:**
```typescript
{
  updates: {
    componentId: string;
    currentVersion: string;   // Version the user has
    latestVersion: string;    // Newest approved version
    changelog: string;        // AI-generated summary of what changed
    updatedAt: string;
  }[];
}
```

**Why POST, not GET:** The list of component IDs could be large. POST avoids URL length limits.

**Client behavior:**
- Poll this endpoint periodically (e.g., on app launch + every 30 minutes while active)
- Show a non-intrusive badge/notification on components that have updates
- User decides whether to pull the update — never auto-update components on the Slate
- When the user pulls an update, the client fetches the new version's source via `GET /api/components/:id/source?version=latest`

**Server behavior:**
- When author A uploads v2, the server creates a new version linked to the same component ID
- The `check-updates` endpoint compares requested version against latest approved version
- Old versions remain accessible (users can pin to a specific version)

### Version history endpoint:
```
GET /api/components/:id/versions
```
Returns a list of all approved versions with changelogs. The client can show this in a "component history" panel.

## 5. Test Render — What It Means

**No headless browser.** The "test render" in our review pipeline is a **transpilation + static analysis check**, not a visual render.

### What "test render passed" actually guarantees:

**Stage 1 (Structural Validation) does:**
1. **TypeScript compilation** — all `.ts`/`.tsx` files compile without errors using a strict tsconfig
2. **Import resolution** — all imports between package files resolve correctly (no broken imports)
3. **React validity** — JSX is syntactically valid, component exports a valid React component (has a default or named export that returns JSX)
4. **Dependency check** — all npm dependencies declared in `manifest.dependencies.npm` are from an allowlist of safe packages

**What it does NOT do:**
- Does not render the component visually (no headless browser, no JSDOM)
- Does not execute runtime code (no `useState` actually runs, no effects fire)
- Does not check that the component "looks right"
- Does not check runtime behavior

**Why not a full render:**
1. **Security risk** — executing untrusted code on the server, even in a sandbox, is a significant attack surface. The review agent checks for malicious patterns without running the code.
2. **Complexity** — a headless browser (Puppeteer/Playwright) adds heavy infrastructure, is slow, and introduces flaky failures.
3. **Diminishing returns** — if TypeScript compiles and the LLM review confirms the logic is sound, runtime rendering failures are rare. The client sandbox is the real test environment.

**What "quality review passed" (Stage 3) guarantees:**
- The LLM has read the code and confirmed it follows React best practices
- UI/logic separation is correct (hook for logic, .tsx for presentation)
- No dead code, no console.logs, proper error handling
- Accessibility basics (semantic HTML, ARIA labels where needed)
- This is a code review, not a test run

**What the client should tell users:**
> "This component has been reviewed for security, code quality, and structure. It has not been visually tested — you may need to adjust styling or behavior to fit your Slate."

This sets the right expectation: the community library is curated for quality and safety, but visual perfection is the user's iterative refinement loop with their local AI agent.
